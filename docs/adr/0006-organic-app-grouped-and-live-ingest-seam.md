# Organic Content: app-grouped response + live ingest/refresh seam with a stubbed source

The Organic Content page (`/dashboard/organic`) was specced as a "near-twin of the Ads Library." Two design calls diverge from that framing enough to record, because a future reader templating off Ads would otherwise expect the wrong shapes.

**1. The API is app-grouped, not a flat creative list.** `GET /api/v1/organic` returns one card per App — the App's hydrated metrics (reusing the same scored-rows path as `/apps`, so REVENUE/INSTALLS/rating aren't re-derived), its store screenshots (Listing media), and a `videos[]` array — paginated **by App** (12/page). The Ads route returns a flat `meta_ads[]` paginated by creative. We grouped because truth groups: the page ranks Apps (`#1…#12`) and shows each App's creator carousel, so the unit of pagination is the App, not the video. The cost is that organic can't literally reuse the Ads route handler; it reuses the App **metric hydration** instead (`getAppListItemsByIds`), which is the valuable part.

**2. Organic data flows through a live ingest + refresh seam, not a static seed.** There is no public creator/UGC video feed wired today — the same capability gap as the dormant Meta Ad Library sync. Rather than hardcode fixtures into a throwaway seed script, the data path is the real one end-to-end: an idempotent ingest **job** (`ingest:organic` → `syncOrganic`) upserts `organic_videos` and moves `lastSeenAt` every run; the page's **"Refresh organic content"** button calls `POST /api/v1/organic/refresh`, which re-runs that same job in-process (the `sync-reviews` pattern); and an `organic-videos` sweep is registered in the freshness scheduler (ADR 0004, `cadenceHours: 12`) so it auto-refreshes on boot catch-up and on cadence — no manual press needed for the surface to stay current. The single concession to the capability gap is isolated behind one swappable object — `stubOrganicSource` (`packages/ingest/src/organic/source.ts`), which emits deterministic representative rows. Swap that one adapter for a real TikTok/Instagram source and nothing downstream — DB, API, page, or refresh — changes. We chose this over a seed script so the surface is a working feature that stays current (scheduled job or Refresh), not a one-time clone of the content. The cost is more scaffolding up front (job + adapter interface + refresh endpoint) than a fixture would need; accepted because a static seed would have to be torn out and rebuilt the day a real feed ships.

Consequence: video tiles render as deterministic gradient + `@handle` overlays (no real video assets to thumbnail), and the seeded batch is intentionally small until correctness is confirmed against truth.
